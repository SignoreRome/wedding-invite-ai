# AGENTS.md

## 1. Контекст проекта

Это проект сайта-приглашения на свадьбу с возможностью сбора RSVP-ответов гостей.

Основные цели проекта:

* сделать сайт, который в первую очередь удобно открывать с телефона;
* сохранить визуальный стиль Windows 95 / Windows XP;
* надежно сохранять ответы гостей;
* запускать приложение на VPS;
* поддерживать проект в простом, понятном и пригодном для продакшена состоянии.

Предпочтительный стек на текущий момент:

* Next.js
* TypeScript
* Tailwind CSS
* Prisma
* SQLite
* Nginx на VPS

---

## 2. Продуктовые требования

При выполнении любых задач агент должен учитывать следующие требования:

* Сайт должен разрабатываться по принципу mobile-first.
* Большая часть гостей будет открывать сайт с телефона, часто из мессенджеров.
* Форма RSVP должна сохранять ответы гостей.
* Приложение должно быть пригодно для запуска на VPS.
* Визуальный стиль должен оставаться в эстетике Windows 95 / XP, но быть адаптирован под современный мобильный UX.
* Реализация должна избегать ненужной сложности.

---

## 3. Общие правила работы агента

### 3.1. Каждая задача выполняется в новой git-ветке

Для каждой новой задачи, исправления, фичи, рефакторинга или изменения документации агент обязан работать в отдельной git-ветке.

Формат именования веток:

* `feature/<краткое-описание>`
* `fix/<краткое-описание>`
* `refactor/<краткое-описание>`
* `docs/<краткое-описание>`
* `chore/<краткое-описание>`

Примеры:

* `feature/rsvp-form-submit`
* `fix/mobile-layout-overflow`
* `docs/update-readme-deploy`

Правила:

* Нельзя выполнять новую задачу напрямую в `main`.
* Перед началом работы нужно проверить текущую ветку.
* Если текущая ветка не соответствует задаче, нужно создать новую и переключиться на нее.
* Одна логическая задача = одна ветка.
* Нельзя смешивать несвязанные изменения в одной ветке.

### 3.2. README.md должен поддерживаться в актуальном состоянии

Любое изменение, которое влияет на установку, запуск, архитектуру, переменные окружения, деплой, модель данных, маршруты или пользовательский сценарий, должно быть отражено в `README.md`.

README нужно обновлять, если изменения затрагивают:

* установку проекта;
* локальную разработку;
* переменные окружения;
* настройку базы данных или миграции;
* команды сборки и запуска;
* деплой на VPS;
* архитектурные решения;
* публичные или административные маршруты;
* flow RSVP;
* экспорт, резервное копирование, хранение данных.

Если задача меняет код, но не требует обновления README, агент должен явно проверить, что обновление документации действительно не нужно.

### 3.3. Изменения должны быть небольшими и сфокусированными

Правила:

* Вносить минимально необходимые изменения.
* Не переписывать большие части проекта без явной необходимости.
* Не ломать существующее рабочее поведение, если задача этого не требует.
* Не совмещать крупный рефакторинг с реализацией новой функциональности без причины.
* Делать diff понятным для ревью.

### 3.4. Нужно сохранять визуальную концепцию

Правила:

* Сохранять стиль Windows 95 / XP.
* Улучшать читаемость и адаптивность, не убивая ретро-идентичность.
* На мобильных устройствах отдавать приоритет удобству, а не декоративной сложности.
* Не превращать проект в типичный современный SaaS-интерфейс.
* Не добавлять тяжелые медиафайлы, сложные анимации и лишние эффекты без явной пользы.

### 3.5. Mobile-first обязателен

Правила:

* Сначала проектировать и реализовывать интерфейс для телефонов.
* Не допускать горизонтального скролла.
* Делать кнопки, поля и элементы управления удобными для касания.
* Не перегружать формы.
* Проверять интерфейс хотя бы на типовых ширинах: 360px, 375px, 390px, 414px.

### 3.6. Архитектура должна оставаться простой

Правила:

* Предпочитать простые технические решения.
* Не добавлять новые зависимости без понятной причины.
* Использовать встроенные возможности Next.js, если их достаточно.
* Сохранять архитектуру пригодной для запуска на одном VPS.
* Считать SQLite базовым вариантом хранения данных, если явно не согласовано иное.

### 3.7. Надежность RSVP-данных критична

Правила:

* Все ответы гостей должны валидироваться на сервере.
* Нельзя полагаться только на клиентскую валидацию.
* Нельзя допускать потери ответов без крайней причины.
* Ошибки должны обрабатываться явно и безопасно.
* Нужно разумно защищаться от дубликатов и некорректных отправок.
* Изменения схемы базы данных должны быть безопасными и продуманными.

### 3.8. Безопасность и контроль доступа обязательны

Правила:

* Нельзя публично открывать административные данные.
* Административные страницы должны быть защищены.
* Нельзя хардкодить секреты.
* Секреты и чувствительные настройки должны храниться в переменных окружения.
* Входные данные должны валидироваться и очищаться.
* Нельзя логировать секреты и чувствительные данные без необходимости.

### 3.9. Нужно постоянно учитывать будущий деплой

Правила:

* Любые решения должны быть совместимы с запуском на VPS.
* Нельзя опираться на serverless-only подходы, если это мешает выбранной архитектуре.
* Пути к файлам, переменные окружения и runtime-поведение должны быть явными.
* Путь хранения SQLite должен быть понятным и постоянным.
* Нельзя хранить важные данные во временных build-директориях.

### 3.10. Код должен оставаться качественным

Правила:

* Предпочитать ясный и читаемый код.
* Корректно использовать TypeScript.
* Избегать `any`, если без него можно обойтись.
* Выносить повторяющиеся части в компоненты и утилиты, если это упрощает код.
* Удалять временный и мертвый код, появившийся в ходе задачи.

## 4. Обязательный workflow для каждой задачи

Для каждой задачи агент должен придерживаться следующего порядка:

1. Понять задачу и ее границы.
2. Проверить текущий git status и текущую ветку.
3. Создать новую ветку под задачу, если это нужно.
4. Изучить релевантные файлы перед изменениями.
5. Реализовать только необходимые изменения.
6. Обновить или проверить типы, валидацию, тесты и связанную логику, если это применимо.
7. Обновить `README.md`, если изменилась документируемая часть проекта.
8. Запустить релевантные проверки.
9. Подготовить краткое резюме изменений и возможных follow-up шагов.

---

## 5. Git-политика

Перед любыми изменениями агент обязан убедиться, что работа изолирована в отдельной ветке.

Типовой порядок команд:

```bash
git status
git branch --show-current
git checkout -b feature/example-task
```

Правила:

* Если под текущую задачу уже существует подходящая ветка, можно продолжать работу в ней.
* Если ветка не соответствует задаче, нужно создать новую.
* Нельзя оставаться в старой ветке для новой несвязанной задачи.
* Нельзя незаметно модифицировать `main`.
* Нельзя смешивать несколько независимых задач в одной ветке.

---

## 6. Политика поддержки README.md

README должен, когда это применимо, содержать:

* назначение проекта;
* используемый стек;
* шаги локального запуска;
* переменные окружения;
* настройку базы данных;
* команды миграций;
* команды разработки;
* production build/run команды;
* заметки по деплою на VPS;
* расположение SQLite-файла;
* информацию о резервном копировании;
* информацию об административном доступе;
* описание RSVP-flow.

Если появляется новая функциональность, агент должен проверить, не требуется ли:

* новая секция в README;
* обновление команд;
* изменение примеров;
* обновление архитектурных заметок.

## 7. Стандарты разработки

### 7.1. UI / Frontend

* Использовать mobile-first подход.
* Предпочитать семантическую HTML-разметку.
* Держать в уме доступность.
* Для форм и интерактивных сценариев иметь состояния loading / success / error.
* По возможности избегать layout shift.
* Делать RSVP-форму простой и удобной на телефоне.

### 7.2. Backend / API

* Валидировать все входные данные.
* Возвращать понятные и一致ные HTTP-статусы.
* Не раскрывать внутренние ошибки сервера наружу.
* Делать обработчики небольшими и читаемыми.
* Использовать явные схемы и типизированные ответы.

### 7.3. База данных

* Менять схему осознанно.
* По возможности предпочитать безопасные и добавочные миграции.
* Избегать разрушительных изменений без явной необходимости.
* Отражать значимые изменения схемы в README.

### 7.4. Деплой

* Сохранять возможность запуска проекта на одном VPS.
* Использовать явную конфигурацию окружения.
* Безопасно организовывать хранение SQLite.
* Документировать изменения, влияющие на деплой.

---

## 8. Чего агенту делать нельзя

Агент не должен:

* выполнять новую задачу напрямую в `main`;
* вносить инфраструктурные изменения без отражения в документации;
* добавлять ненужные зависимости;
* ломать ретро-стиль в пользу шаблонного современного редизайна;
* переусложнять проект ради гипотетического масштабирования;
* вносить ломающие изменения без явного указания и описания;
* пропускать обновление README, если изменилось поведение системы;
* делать тихие изменения схемы базы данных без учета хранения и миграций.

---

## 9. Поведение агента при неоднозначной задаче

Если требования неполные или неоднозначные, агент должен:

* сохранять текущие архитектурные решения;
* выбирать самый простой вариант реализации, совместимый с проектом;
* не создавать абстракции “на будущее” без реальной необходимости;
* явно фиксировать свои допущения в итоговом описании изменений.

---

## 10. Definition of Done

Задача считается завершенной только если:

* работа выполнена в корректной git-ветке;
* реализовано требуемое поведение;
* затронутая документация обновлена;
* релевантные проверки запущены;
* не изменены лишние файлы без причины;
* результат не противоречит целям проекта и визуальной концепции.

---

## 11. Приоритеты проекта

Если приходится выбирать между несколькими компромиссами, приоритеты такие:

1. корректность и сохранность RSVP-данных;
2. удобство использования на телефоне;
3. простота деплоя на VPS;
4. поддерживаемость кода;
5. сохранение стилистики Windows 95 / XP;
6. второстепенные улучшения.
